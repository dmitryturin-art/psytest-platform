# Дизайн: Очистка и рефакторинг PsyTest Platform

**Дата:** 2026-06-22
**Статус:** Одобрен пользователем, готов к плану реализации
**Ветка:** `refactor/technical-debt-phase1`
**Подход:** Гибрид C+A (контракт-первый + корректность)

---

## 1. Контекст и цели

**Проект:** PsyTest Platform — модульная PHP-платформа психологических тестов (PHP 8.1+, MySQL, Twig 3, DomPDF 3). Заказчик — клинический психолог, использует платформу для своей практики.

**Цель работы:** Общий аудит и чистота кодовой базы — удалить мусор, разнести ответственность, покрыть расчёты тестами. Без привязки к конкретному релизу, но с прицелом на корректность и расширяемость.

### Видение продукта

- Модульная платформа: каждый тест = изолированный модуль со своим расчётом и выводом, общая оболочка и дизайн.
- Своя коллекция тестов (без рекламы), AI-интерпретация — платная/админская функция на будущее (инфраструктура уже есть, откладывается).
- Корректность расчётов — верификация по методике Собчик + эталон psytests.org (значения).

### Коллекция тестов (первый этап)

| Тест | Тип | Приоритет |
|------|-----|-----------|
| СМИЛ/MMPI | Личностный, T-баллы, валидация | Главный, существует |
| BAI (тревога Бека) | Суммирующий | Существует |
| BDI (депрессия Бека) | Суммирующий | Новый |
| HADS (тревога+депрессия) | Суммирующий, 2 подшкалы | Новый |

16PF и Люшер — позже (нет исходников/норм, либо принципиально другая механика).

### Гендерные формы СМИЛ

Сейчас реализованы 2 формы (муж/жен) — и в текстах вопросов, и в нормах. Архитектура закладывается сразу под 4 формы (муж/жен/мальчики/девочки), реализуются 2. Подростковые нормы добавляются позже, когда будут извлечены из Собчик.

## 2. Решение по стеку

**Стек сохраняется (PHP + MySQL + Twig + DomPDF).** Аргументы:

- Все выявленные проблемы (дубли JSON, god-класс, отсутствие тестов) — не стековые, смена языка их не решит.
- Заказчик — не разработчик: PHP-монолит деплоится на любой shared-хостинг, переписывание на Node/Python потребует VPS и runtime-настройки.
- Смена стека = выбросить проверяемую логику скоринга и пере-реализовать СМИЛ вслепую — недопустимо при текущем недоверии к расчётам.

**Усиление toolchain** (dev-зависимости в composer.json):

| Инструмент | Назначение |
|------------|------------|
| PHPUnit | Тесты расчётов на эталонных значениях |
| PHPStan (level 6+) | Статический анализ, ловит баги типов до запуска |
| PHP CS Fixer | Единый стиль кода (PSR-12) |
| Phinx | Версионные миграции БД вместо ручного schema.sql |

**Не вводится:** полноценный фреймворк (Laravel/Symfony) — текущий Router + Twig достаточны для монолита этого масштаба.

**Убрать следы `test-smil.php`** (упоминается в коммитах, файла нет) из доков.

### Поток 2 — Модульный контракт

**Проблема:** `renderResults()` в `TestModuleInterface` заставляет каждый модуль генерировать HTML-строки вручную (~700 строк в SmilModule). Это смешивает данные и представление.

**Решение — разделение слоёв:** модуль = расчёт и данные, шаблон Twig = представление.

Страница результата = общая **оболочка** (header, шапка теста, дата, дисклеймер, кнопка PDF, навигация, подвал — одинакова для всех) + **блоки** (тут тесты отличаются: состав и порядок + данные).

Переиспользуемые twig-компоненты (`profile-chart.twig`, `scales-table.twig`, `score-badge.twig`) рисуют блоки по данным. Общая `result-layout.twig` собирает страницу.

**Контракт:**

```
TestModuleInterface
├── getMetadata(): array
├── getQuestionsFor(Form): array    — вопросы с учётом формы респондента
├── score(answers, Form): ResultData — ЕДИНЫЙ расчёт, возвращает структуру
├── getNorms(): array               — нормы/шкалы модуля
├── supportsPairMode(): bool
└── getTemplates(): array           — какие twig-шаблоны/блоки использовать
```

Ключевые изменения:
1. `renderResults()` убирается из контракта. Модуль возвращает структурированный `ResultData` (секции: raw_scores, t_scores, validity, profile, interpretation). Общий шаблон рисует HTML.
2. Явная «форма» респондента (`male`/`female`/`boy`/`girl`) вместо неявного gender в answers. Архитектура под 4 формы сразу, реализуются 2.
3. `ResultData` — единый контракт вывода. BDI/HADS не пишут свой HTML, только считают и отдают данные.
4. `getCustomJavaScript()` остаётся для особых случаев (типа Люшера) — как исключение.

#### Разнообразие механик тестов (явное требование к контракту)

Тесты в коллекции принципиально разные по механике ввода и вывода. Контракт обязан это поддерживать, не навязывая единый формат:

**Механика ввода (форматы ответов):**
- СМИЛ/BDI/HADS — Да/Нет или Лайкерт (стандартные radio/шкала).
- Люшер — выбор и упорядочивание цветов (визуальный, drag-n-drop, нужен свой JS + canvas).
- ТАТ — ввод текста (свободный рассказ по картинкам).
- Опросники с картинками-стимулами.

**Механика вывода (страница результата):**
- Скоринговые тесты (СМИЛ, BDI, HADS) — стандартные блоки (профиль, таблицы, бейджи, интерпретация).
- Люшер — своя визуализация (последовательность цветов + интерпретация структуры).
- ТАТ — качественный анализ текста (совсем другая структура результата).

**Парный режим (flow двух сессий):**
Например, опросник супружеской удовлетворённости (Лазарус): оба супруга отвечают **в разных сессиях**, затем ответы связываются **по коду**, система формирует общий результат сравнения + интерпретацию, которую можно прислать психологу.

Архитектурно контракт это поддерживает через два режима:

1. **Стандартный режим** (скоринговые тесты): модуль отдаёт `ResultData` + объявляет список блоков → общая оболочка рисует страницу из переиспользуемых twig-компонентов. Это убирает HTML из модулей для большинства тестов.

2. **Кастомный режим** (принципиально другая механика — Люшер, ТАТ, парные): модуль через `getTestTemplate()` / `getResultTemplate()` отдаёт полностью свой twig-шаблон, а `getCustomJavaScript()` — свою логику ввода. Модуль по-прежнему отдаёт данные, но отрисовку полностью контролирует сам. `renderResults()` в PHP-коде не возвращается — шаблон всегда в Twig.

Парный flow (2 сессии + связывание кодом + общий результат) — это инфраструктурная задача поверх контракта (управление сессиями, код-связывание в БД). **Реализация откладывается** (не входит в этап), но контракт `supportsPairMode()` / `comparePairResults()` уже заложен и не будет этому препятствовать. Схема БД и контроллеры проектировались с учётом расширения под парные сессии.

### Поток 3 — Калькуляторы СМИЛ + тесты (ядро доверия)

#### КРИТИЧЕСКАЯ НАХОДКА: ключи разметки неверны

Сверка числа пунктов по шкалам с эталоном Собчик показала, что разметка «вопрос→шкала→направление» в `questions-566-full.json` **неверна** по большинству шкал:

| Шкала | В файле | Эталон Собчик | Направление в файле | Должно быть |
|-------|---------|---------------|---------------------|-------------|
| L (ложь) | 40 | 15 | все прямые (1) | 15, все обратные |
| F (достоверность) | 40 | 64 | все прямые | 64, все прямые |
| K (коррекция) | 26 | 30 | прямые | 30, обратные |
| 1 (ипохондрия) | 42 | 33 | смешанное | 33 |
| 8 (шизофрения) | 40 | 78 | все прямые | 78 |
| 0 (интроверсия) | 49 | 70 | смешанное | 70 |

**Следствие:** формула T-баллов `convertToTScores()` верна (проверена на эталоне raw→T), но **сырые баллы вычисляются неверно**, потому что ключи некорректны. Совпадение с эталоном при ручной проверке было кажущимся — там брались готовые raw, а не вычислялись из ответов.

#### Этап 3a — Верификация ключей (первоочередная задача)

Источники для восстановления:
1. Книги Л.Н. Собчик и И.Л. Соломина в библиотеке заказчика (ключи: номера пунктов + направление для каждой шкалы).
2. Бланк psytests.org (`smilF-bl.html`) — номера вопросов и тексты.
3. Образец результата psytests.org — end-to-end проверка (вычисленные raw должны совпасть с эталонными для конкретного ответа).

Результат: корректный `questions-566-full.json` с правильными `scale` и `direction` для всех 13 шкал.

#### Этап 3b — Калькуляторы + PHPUnit

Разбивка скоринга на 4 чистых класса в `modules/smil/Scoring/`:

```
modules/smil/Scoring/
├── RawScoreCalculator.php          — ответы → сырые баллы (13 шкал)
├── TScoreCalculator.php            — сырые баллы + пол → T-баллы (линейная формула + K-коррекция)
├── ValidityAssessor.php            — L/F/K/QC/? → валидность протокола
└── AdditionalScalesCalculator.php  — 200+ доп. шкал
```

`SmilModule` становится тонким оркестратором: вызывает калькуляторы, собирает `ResultData`, отдаёт шаблону. Минус ~700 строк HTML + ~400 строк логики.

**PHPUnit-сетка:**
- Эталонные: конкретные raw→T пары из psytests.org (L=4→T=49, K=19→T=63, шкала 1=9→T=63 с K-коррекцией, и т.д.).
- Краевые: raw=0, maxRaw, K-коррекция=0, шкала «?»=0/40/60/70, QC=19/20.
- Инварианты: монотонность, диапазоны T (20–120).
- End-to-end: полный набор ответов → сырые баллы → сверка с образцом psytests.org.

Формула расчёта не меняется (она верна) — только переносится в чистые классы и покрывается тестами. Но ключи восстанавливаются (этап 3a), иначе тесты raw→T не пройдут.

### Поток 4 — Новые тесты BDI + HADS

Проверка контракта на тестах проще СМИЛ + расширение коллекции.

- **BDI (депрессия Бека):** 21 вопрос, 4 варианта по тяжести (0–3), сумма → уровень (мин/лёгкая/умеренная/тяжёлая).
- **HADS (госпитальная шкала):** 14 вопросов, 2 подшкалы (тревога A + депрессия D) по 7 пунктов, каждый 0–3. A=0–21, D=0–21 → уровни. PDF в библиотеке заказчика.

Каждый тест = `metadata.json` + `questions.json` + один класс модуля (только расчёт), вывод через общие twig-блоки. PHPUnit-тесты на суммирование и пороги.

Порядок: BDI первым (структурно ближе к BAI), HADS вторым.
